Reserva atômica recomendada (RESEARCH.md seção 4 — opção MVP via $transaction):
```typescript
await fastify.prisma.$transaction(async (tx) => {
  const user = await tx.user.findUnique({ where: { id: userId } })
  if (!user || user.creditBalance < quantity) {
    throw { statusCode: 400, message: 'Créditos insuficientes' }
  }
  await tx.user.update({ where: { id: userId }, data: { creditBalance: { decrement: quantity } } })
  await tx.order.create({ data: { ... } })
  await tx.creditTransaction.create({ data: { ... } })
})
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Cliente → POST /orders | scheduledDate e quantity são input não confiável |
| Cliente → POST /users/push-token | playerId é string arbitrária do cliente |
| Service → prisma.$transaction | Atomicidade de crédito depende do transaction |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03-01 | Elevation of Privilege | POST /orders | mitigate | preHandler: fastify.authenticate; userId extraído do JWT, nunca do body |
| T-04-03-02 | Tampering | scheduledDate no POST /orders | mitigate | Validar scheduledDate >= tomorrow no service; rejeitar datas no passado com 400 |
| T-04-03-03 | Tampering | quantity no POST /orders | mitigate | CreateOrderSchema: z.number().int().min(1).max(20) via Zod |
| T-04-03-04 | Tampering | Double-spend de créditos | mitigate | prisma.$transaction garante consistência no MVP; documentado que MVP não implementa $runCommandRaw (risco baixo com poucos usuários) |
| T-04-03-05 | Spoofing | POST /users/push-token salva para outro usuário | mitigate | userId é sempre request.user.id do JWT — body não aceita userId externo |
| T-04-03-06 | Tampering | playerId de formato inválido | accept | z.string().min(1) valida presença; formato OneSignal não precisa ser validado estruturalmente |
  | T-04-SC | Tampering | npm installs | mitigate | Pacotes auditados em 04-RESEARCH.md |
