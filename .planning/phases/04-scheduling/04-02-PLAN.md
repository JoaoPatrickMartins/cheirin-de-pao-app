## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Cliente → PUT /schedules/me | Input não confiável — weeklyQty e deliveryTime vêm do cliente |
| Service → MongoDB Atlas | Writes de creditBalance devem ser atômicos |
| Service → OneSignal API | Push enviado a player_ids vindos do banco — podem estar expirados |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-02-01 | Elevation of Privilege | GET/PUT /schedules/me | mitigate | preHandler: fastify.authenticate em todas as rotas; buscar schedule via userId === request.user.id |
| T-04-02-02 | Tampering | weeklyQty no PUT /schedules/me | mitigate | ScheduleBodySchema: cada dia z.number().int().min(0).max(12) — rejeita valores negativos e acima de 12 |
| T-04-02-03 | Tampering | deliveryTime no PUT /schedules/me | mitigate | z.enum(['06:30','07:00','07:30','08:00']) — rejeita qualquer valor fora da lista |
| T-04-02-04 | Tampering | creditBalance race condition em createDailyOrders | accept | Cron é sequencial por usuário — risco de race condition muito baixo no MVP; documentado em RESEARCH.md pitfall 2 |
| T-04-02-05 | Spoofing | push via player_id expirado | accept | Falha silenciosa (log warn) não propaga erro — comportamento esperado conforme RESEARCH.md seção 2 |
| T-04-SC | Tampering | npm/pip/cargo installs | mitigate | slopcheck concluído em 04-RESEARCH.md — node-cron e @onesignal/node-onesignal aprovados |
</threat_model>
