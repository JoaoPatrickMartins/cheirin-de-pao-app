<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → POST /auth/otp/send | Phone/email submitted; API validates format and creates/reuses OtpCode |
| Browser → POST /auth/otp/verify | 4-digit code + deviceId; returns session token if valid |
| apiFetch → all API endpoints | All requests include X-Device-Id for device binding |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-12 | Tampering | OTP input — non-numeric injection | mitigate | OtpInput rejects non-digits via /^\d?$/ guard before state update; API also validates code.length === 4 via VerifyOtpSchema |
| T-02-13 | Information Disclosure | apiFetch injects auth_token from localStorage | accept | Token in localStorage per D-02 (XSS risk acknowledged); app does not render external HTML |
| T-02-14 | Tampering | Login form — rapid double-submit (Pitfall 5 analog) | mitigate | "Enviar código" button disabled and shows loading state on first tap; OTP verify auto-fires via onComplete (no double-submit possible on OTP step) |
| T-02-15 | Spoofing | userId returned from POST /auth/otp/send and sent back in POST /auth/otp/verify | accept | API validates that userId matches the OtpCode record before creating session; client spoofing userId without a valid matching OtpCode returns 401 |
</threat_model>
