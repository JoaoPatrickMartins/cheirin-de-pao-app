<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client browser → POST /auth/register | Untrusted user input: name, CPF, phone, email, condominiumId, apartment |
| Client browser → POST /auth/otp/verify | Untrusted: 4-digit code + deviceId; session token is created here and returned once |
| API → Zenvia SMS API | Outbound: phone number + OTP code in plaintext; protected only by HTTPS in transit |
| API → Resend email API | Outbound: email address + OTP code; protected only by HTTPS in transit |
| Session token in DB | Token stored as sha256 hash — raw token only returned once on POST /auth/otp/verify |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | POST /auth/otp/verify — OTP brute force | mitigate | 10-min expiresAt enforced in auth.service.ts (check expiresAt < new Date()); usedAt set immediately on first use preventing replay; 10,000 possible codes × 10 min TTL limits window |
| T-02-02 | Information Disclosure | Session token storage | mitigate | sha256(rawToken) stored in DB; plaintext returned to client only once on verifyOtp response |
| T-02-03 | Spoofing | Device ID mismatch (X-Device-Id header) | accept | device_id is a convenience signal per D-08, not a security control; session token hash is the primary auth; spoofing device_id without the token provides no access |
| T-02-04 | Elevation of Privilege | POST /auth/couriers — admin-only | mitigate | preHandler: [fastify.authenticate] + role === ADMIN check in controller before creating COURIER user |
| T-02-05 | Tampering | Admin bootstrap — env var injection | accept | Admin only created if no ADMIN role exists (idempotent); bootstrap reads env vars at server startup, not at runtime |
| T-02-06 | Information Disclosure | OTP code in SMS/email | accept | HTTPS in transit to Zenvia/Resend; dev mode never sends to external services (OTP_DEV_CODE bypass); code is 4-digit (acceptable for UX-focused auth at this scale) |
| T-02-07 | Tampering | Input validation — CPF, phone, email | mitigate | Zod schema parses all request bodies; CpfSchema strips /\D/g and validates módulo-11; invalid input returns 400 before reaching service layer |
| T-02-SC | Tampering | npm install resend | mitigate | Package legitimacy audit in RESEARCH.md confirms resend [OK] — approved. No checkpoint needed. |
</threat_model>
