API endpoint contracts (from Plan 02):
```
POST /auth/register
  Body: { name, cpf, birthDate, phone?, email?, channel: 'sms'|'email', condominiumId, apartment, block? }
  Response 201: { userId: string }  // returns userId to use in subsequent OTP send

GET /condominiums
  Response 200: [{ id, name, type: 'SINGLE_ENTRANCE'|'BLOCKS', neighborhood }]

POST /auth/otp/send
  Body: { phone?: string, email?: string }
  Response 200: { userId: string }

POST /auth/otp/verify
  Body: { userId, code (4 chars), deviceId }
  Response 200: { token, user: { id, role, name } }

POST /auth/couriers (admin auth required — apiFetch injects token)
  Body: { name, cpf, phone?, email? }
  Response 201: { id, name, role: 'COURIER' }
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → POST /auth/register | Untrusted: name, CPF, phone, email, apartment, condominiumId, block |
| Browser → POST /auth/couriers | Admin-authenticated: name, CPF, phone, email of new courier |
| GET /condominiums | Public endpoint — no authentication needed; returns only public condo info |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-16 | Tampering | POST /auth/register — duplicate registration (Pitfall 5) | mitigate | "Enviar código de confirmação" button disabled on tap + loading state; API returns 409 on duplicate; frontend shows "Esse CPF já tem uma conta." error message |
| T-02-17 | Tampering | CPF validation bypass via formatting | mitigate | Frontend strips /\D/g before sending; CpfSchema on API also strips + validates módulo-11 (per AUTH-02); double validation per Architectural Responsibility Map |
| T-02-18 | Elevation of Privilege | POST /auth/couriers without admin auth | mitigate | API preHandler: fastify.authenticate + role === ADMIN check (Plan 02). CourierRegisterScreen also checks user.role === 'ADMIN' client-side and redirects if not. |
| T-02-19 | Tampering | Condominium ID spoofing in registration | accept | condominiumId is validated server-side (API checks prisma.condominium.findUnique); if invalid, register returns 400. Frontend loads condos from API so options are always valid. |
| T-02-20 | Information Disclosure | AUTH-03 block/tower conditional — block sent when condo is not BLOCKS type | mitigate | Block field only enabled when condoType === 'BLOCKS' in UI; API service validates block requirement after fetching condo type from DB. |
</threat_model>
