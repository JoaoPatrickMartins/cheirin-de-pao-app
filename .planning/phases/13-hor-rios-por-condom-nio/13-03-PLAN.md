Nova lógica de `processCutoff()`:

1. Calcular `nowHHMM` no fuso BRT usando `Intl.DateTimeFormat` (mesmo padrão do método atual).

2. Buscar todos os condomínios ativos com seus deliverySlots:
   ```typescript
   const condominiums = await this.prisma.condominium.findMany({
     where: { isActive: true },
   })
   ```

3. Para cada condomínio, filtrar slots ativos cujo `cutoffTime === nowHHMM`.

4. Se nenhum condomínio/slot corresponde ao horário atual: retornar sem ação.

5. Para cada par (condomínio, slot) em corte:
   a. Calcular tomorrowStart/tomorrowEnd BRT (mesmo padrão do código atual).
   b. Buscar Orders de amanhã DESTE condomínio: `where: { condominiumId: condo.id, scheduledDate: { gte, lte }, status: { not: 'CANCELLED' } }`.
   c. Buscar clientes do condomínio com oneSignalPlayerId: `where: { role: 'CLIENT', condominiumId: condo.id, oneSignalPlayerId: { not: null }, isBlocked: false }`.
   d. Filtrar clientes sem Order amanhã.
   e. Enviar push com mensagem mencionando o slot: slot.name === 'manha' → "Prazo de pãezinhos da manhã de amanhã encerrando!" / slot.name === 'tarde' → "Prazo de pãezinhos da tarde de amanhã encerrando!". Push best-effort via OneSignal (mesmo padrão existente com `osClient`).

ATENÇÃO — Pitfall 3 do RESEARCH: não usar loop O(N²). A busca de Orders e de clientes por condomínio já é batched por condomínio, não por usuário individual. Manter o padrão eficiente.

Nova rota:
```
GET /client/condominium/slots
preHandler: [fastify.authenticate]
```

Lógica do handler:
1. Extrair `user` do JWT via `request.user`.
2. Se `user.condominiumId` não existir: retornar 404 `{ message: 'Usuário sem condomínio vinculado' }`.
3. Buscar o condomínio: `prisma.condominium.findUnique({ where: { id: user.condominiumId }, select: { deliverySlots: true } })`.
4. Se condomínio não encontrado: retornar 404 `{ message: 'Condomínio não encontrado' }`.
5. Filtrar slots ativos: `condo.deliverySlots.filter((s) => s.isActive)`.
6. Retornar array de slots ativos (sem outros dados do condomínio).

ATENÇÃO de segurança — NÃO aceitar `condominiumId` de query param ou body. Usar APENAS `user.condominiumId` do JWT (RESEARCH Pitfall 5 — IDOR).

ATENÇÃO — Verificar se `fastify.authenticate` já está disponível no contexto do plugin `condominiumsRoute`. Se o arquivo usar um padrão diferente de autenticação, seguir o padrão do módulo mais próximo (ex: `orders.route.ts` ou `schedules.route.ts`).
